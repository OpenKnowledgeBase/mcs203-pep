Page Replacement Algorithms

Page replacement algorithms decide which memory pages to swap out when a new page needs to be loaded into memory. The goal is to minimize page faults which can significantly impact system performance.

Page Reference String

1, 2, 3, 4, 2, 1, 5, 6, 2, 1, 2, 3, 7, 6, 3, 2, 1, 2, 3, 6

(Assuming 3 frames, all initially empty)

FIFO Replacement

Replaces the oldest page in memory.

Page Faults: 15

1: [1] (Fault)

2: [1, 2] (Fault)

3: [1, 2, 3] (Fault)

4: [2, 3, 4] (Fault, 1 replaced)

...

LRU Replacement

Replaces the least recently used page.

Page Faults: 12

1: [1] (Fault)

2: [1, 2] (Fault)

3: [1, 2, 3] (Fault)

4: [2, 3, 4] (Fault, 1 replaced)

...

Optimal Replacement

Replaces page not used for longest time in future.

Page Faults: 11

1: [1] (Fault)

2: [1, 2] (Fault)

3: [1, 2, 3] (Fault)

4: [2, 3, 4] (Fault, 1 replaced)

...

Algorithm Comparison

Algorithm Page Faults Implementation Practical Use
FIFO 15 Simple (queue) Basic systems
LRU 12 Complex (counters/stack) General purpose
Optimal 11 Not practical (requires future knowledge) Benchmarking only

Key Takeaways

  • Optimal provides the fewest page faults but cannot be implemented in practice
  • LRU is a good approximation of optimal and is commonly used
  • FIFO is simple but suffers from Belady's anomaly (more frames can lead to more faults)
  • The choice depends on the trade-off between implementation complexity and performance

Disk Scheduling Algorithms

Disk scheduling algorithms determine the order in which disk I/O requests are serviced. The goal is to minimize seek time and maximize disk throughput.

Disk Queue Example

FCFS (First-Come, First-Served)

Services requests in the order they arrive.

Total Movement: 3792 tracks

874 → 692 (182)

692 → 475 (217)

475 → 105 (370)

...

SSTF (Shortest Seek Time First)

Services the request with minimum seek time from current head position.

Total Movement: 1022 tracks

123 → 105 (18)

105 → 12 (93)

12 → 376 (364)

376 → 475 (99)

...

SCAN (Elevator Algorithm)

Moves the head in one direction, servicing requests, then reverses direction.

Total Movement: 1022 tracks

123 → 12 (111, towards 0)

12 → 923 (911, reverse direction)

Services all requests in path

LOOK

Similar to SCAN but only goes as far as the last request.

Total Movement: 1022 tracks

123 → 12 (111, towards 0)

12 → 923 (911, reverse direction)

Services all requests in path

Algorithm Comparison

Algorithm Total Movement Advantages Disadvantages
FCFS 3792 tracks Simple, fair Poor performance
SSTF 1022 tracks Minimizes seek time Starvation possible
SCAN 1022 tracks No starvation, good performance Goes to disk end
LOOK 1022 tracks No starvation, better than SCAN More complex

Key Takeaways

  • SSTF provides the best performance but can lead to starvation
  • SCAN and LOOK provide a good balance between performance and fairness
  • FCFS is simple but has poor performance
  • The choice depends on the workload and performance requirements

Banker's Algorithm for Deadlock Avoidance

The Banker's algorithm is a resource allocation and deadlock avoidance algorithm that tests for safety by simulating the allocation of predetermined maximum possible amounts of all resources.

Example Scenario

Allocation Matrix

Process A B C
P0 0 1 0
P1 2 0 0
P2 3 0 2
P3 2 1 1
P4 0 0 2

Max Matrix

Process A B C
P0 7 5 3
P1 3 2 2
P2 9 0 2
P3 2 2 2
P4 4 3 3

Safety Algorithm Steps

  1. Let Work and Finish be vectors of length m and n respectively. Initialize Work = Available and Finish[i] = false for i = 0, 1, ..., n-1
  2. Find an index i such that both
    1. Finish[i] == false
    2. Need_i ≤ Work
  3. If no such i exists, go to step 5
  4. Work = Work + Allocation_i; Finish[i] = true; go to step 2
  5. If Finish[i] == true for all i, then the system is in a safe state

C Implementation

#include <stdbool.h>
#include <stdio.h>

#define P 5 // Number of processes
#define R 3 // Number of resources

void calculateNeed(int need[P][R], int max[P][R], int allot[P][R]) {
    for (int i = 0; i < P; i++)
        for (int j = 0; j < R; j++)
            need[i][j] = max[i][j] - allot[i][j];
}

bool isSafe(int processes[], int avail[], int max[][R], int allot[][R]) {
    int need[P][R];
    calculateNeed(need, max, allot);

    bool finish[P] = {0};
    int safeSeq[P];
    int work[R];
    for (int i = 0; i < R; i++)
        work[i] = avail[i];

    int count = 0;
    while (count < P) {
        bool found = false;
        for (int p = 0; p < P; p++) {
            if (!finish[p]) {
                int j;
                for (j = 0; j < R; j++)
                    if (need[p][j] > work[j])
                        break;
                
                if (j == R) {
                    for (int k = 0; k < R; k++)
                        work[k] += allot[p][k];
                    
                    safeSeq[count++] = p;
                    finish[p] = true;
                    found = true;
                }
            }
        }
        
        if (!found) {
            printf("System is not in safe state\n");
            return false;
        }
    }
    
    printf("System is in safe state.\nSafe sequence is: ");
    for (int i = 0; i < P; i++)
        printf("%d ", safeSeq[i]);
    printf("\n");
    return true;
}

int main() {
    int processes[] = {0, 1, 2, 3, 4};
    int avail[] = {3, 3, 2};
    
    int max[P][R] = {
        {7, 5, 3},
        {3, 2, 2},
        {9, 0, 2},
        {2, 2, 2},
        {4, 3, 3}
    };
    
    int allot[P][R] = {
        {0, 1, 0},
        {2, 0, 0},
        {3, 0, 2},
        {2, 1, 1},
        {0, 0, 2}
    };
    
    isSafe(processes, avail, max, allot);
    return 0;
}

Key Takeaways

  • The Banker's algorithm prevents deadlock by ensuring the system always remains in a safe state
  • It requires knowledge of maximum resource needs in advance
  • The safety algorithm checks if there's at least one sequence where all processes can complete
  • While theoretically sound, it's rarely used in practice due to its assumptions and overhead

Windows 11 Operating System

Windows 11 introduces several enhancements and innovations in terms of process management, memory management, I/O management, file management, and security and protection.

Core Components

Process Management

  • Employs preemptive multitasking approach
  • Uses Windows Scheduler to prioritize processes
  • Considers process priority, CPU time requested, and power mode
  • Ensures no single application can monopolize system resources

Memory Management

  • Uses virtual memory system with paging
  • Memory Manager handles memory isolation between processes
  • Allows hard drive space to extend RAM when needed
  • Maintains system stability and security

I/O Management

  • Uses device drivers to communicate with hardware
  • More efficient handling of device interruptions
  • Supports a wide range of devices
  • Drivers serve as translation layer between hardware and software

File Management

  • Handled by NTFS (New Technology File System)
  • Supports large volumes, long file names, security features
  • Faster file search capabilities
  • Better integration with cloud storage services

Security and Protection

  • Hardware-based isolation: Secures critical parts of the OS by isolating them
  • Windows Hello: Provides secure biometric authentication
  • Windows Defender: Comprehensive protection against malware and phishing
  • Improved firewall: Enhanced network security features
  • Secure Boot: Prevents unauthorized operating systems from loading
  • TPM 2.0 requirement: Hardware-based security for encryption keys

Key Takeaways

  • Windows 11 provides robust process and memory management for efficient resource utilization
  • Enhanced I/O management supports a wide range of devices with efficient handling
  • NTFS file system offers advanced features and cloud integration
  • Security has been significantly bolstered with hardware-based features and improved protections

LuneOS Mobile Operating System

LuneOS is a mobile operating system built upon the legacy of webOS. Its development is spearheaded by the webOS Ports community, with the aim of creating a fully open-source operating system for mobile devices.

Features of LuneOS

User Interface

Inherits the card-based multitasking interface from webOS, enabling intuitive app switching and management.

Connectivity

Supports Wi-Fi, Bluetooth, and cellular data for comprehensive wireless connectivity options.

Notifications

Innovative non-intrusive notification system accessible with simple gestures.

Open Web Standards

Strong emphasis on HTML5, CSS, and JavaScript, making it attractive for web developers.

System Management

Process Management

  • Uses Linux kernel at its core
  • Preemptive multitasking environment
  • Efficient CPU time allocation to processes
  • Priority assignment based on process state

Memory Management

  • Handled by Linux kernel
  • Uses virtual memory
  • Storage space extends RAM
  • Process isolation for stability

Security Management

  • Process isolation
  • Permissions system
  • Regular updates
  • SELinux for additional security

Key Takeaways

  • LuneOS continues the webOS legacy with a modern open-source approach
  • Features an intuitive card-based interface and innovative notification system
  • Strong support for web standards makes it developer-friendly
  • Leverages Linux kernel capabilities for robust system management
  • Emphasizes security through isolation, permissions, and SELinux

Key Formulas and Values

Scheduling Algorithms

  • Turnaround Time = Completion Time - Arrival Time
  • Waiting Time = Turnaround Time - Burst Time
  • Response Ratio = (Waiting Time + Burst Time) / Burst Time

Page Replacement

  • Page Fault Rate = Number of Page Faults / Total Memory Accesses
  • Effective Access Time: (1-p) × memory access + p × page fault time
